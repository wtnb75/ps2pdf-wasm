# Ghostscript wasm PoC Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build Ghostscript's `gs` binary as wasm via emscripten, restricted to the `pdfwrite` device, and prove that Node.js can run it to convert a PostScript file into a PDF.

**Architecture:** Cross-compile `ghostscript-10.07.1/` with `emconfigure`/`emmake` (configure already supports cross-compiling: it auto-detects a native `CCAUX=gcc` for host-side build tools like `genconf`/`echogs`/`mkromfs` while using `emcc` for the target). The build is linked with `-sNODERAWFS=1`, which makes the wasm binary's filesystem calls operate directly on the real host filesystem under Node (instead of the default in-memory MEMFS) — so Ghostscript can read its `Resource/`/`lib/` init files and write its PDF output using ordinary host paths, with no `--preload-file` packaging step needed. The resulting `./bin/gs` (a Node-runnable JS file plus `./bin/gs.wasm`) is copied into `poc/`, where a small Node script runs it against a test PostScript file (pointing `GS_LIB` at the source tree's `lib`/`Resource/Init` directories) and checks that the output is a valid PDF.

**Note on deviation from the design doc:** The design doc describes `--preload-file` + `MODULARIZE=1` + `callMain()`. During plan research, `-sNODERAWFS=1` + direct `execFileSync('node', ['bin/gs', ...args])` was found to be simpler and avoids guessing MEMFS path mappings for both input resources and output files. The observable behavior and success criteria (Node converts `test.ps` to a PDF starting with `%PDF`) are unchanged.

**Tech Stack:** emscripten 3.1.69 (emcc/emconfigure/emmake, already installed), GNU make, Node.js, the bundled `ghostscript-10.07.1` autotools build.

## Verified facts (from exploration, do not re-derive)

- `emconfigure ./configure --host=wasm32-unknown-emscripten --with-drivers=pdfwrite --disable-dynamic --disable-contrib --disable-cups --disable-fontconfig --disable-gtk --without-x --without-libtiff --without-pdftoraster --without-ijs --without-tesseract` succeeds (exit 0) and produces a `Makefile` with `CC=/usr/share/emscripten/emcc`, `CCAUX=gcc`, `CXX=/usr/share/emscripten/em++`, pointer size 4 (wasm32).
  - `--without-jbig2dec` MUST NOT be passed — pdfwrite requires a JBIG2 decoder and configure errors with "No JBIG2 decoder available, required for PDF support".
  - `--without-tesseract` is required to avoid compiling the ~270-file Tesseract OCR sources, which are not needed for `pdfwrite` and take a very long time to build.
- `emmake make gs` compiles hundreds of `.c`/`.cpp` files successfully with `emcc`/`em++` (verified up to 490 object files with no errors before the exploratory build was stopped early to switch to the `--without-tesseract` config).
- The final binary target is `./bin/gs` (from `BINDIR=./bin` and `GS=gs`, `XE=` empty). The link step uses `GS_LDFLAGS=$(LDFLAGS)=$(AC_LDFLAGS) $(XLDFLAGS)`, so extra emcc link flags can be injected via the `XLDFLAGS` make variable without editing any Makefile.
- emcc, when `-o` has no recognized extension (e.g. `./bin/gs`), emits a Node-runnable JS launcher at that path plus `./bin/gs.wasm`. The JS launcher can be run directly as `node ./bin/gs <args...>`, forwarding `process.argv` to `main(argc, argv)`.
- `-sNODERAWFS=1` is a standard emscripten flag for Node targets that replaces MEMFS with direct calls into Node's `fs`, so file paths passed on the command line (e.g. `-sOutputFile=/abs/path/out.pdf`, or a PostScript input path) and environment variables read via `getenv` (e.g. Ghostscript's `GS_LIB`) resolve to real files on disk, exactly like a natively-built `gs`.

- [ ] **Step 2: Create the build script**

Create `poc/build.sh`:

```bash
#!/bin/bash
set -euo pipefail

ROOT="$(cd "$(dirname "${BASH_SOURCE[0]}")/.." && pwd)"
GS_SRC="$ROOT/ghostscript-10.07.1"
POC="$ROOT/poc"

cd "$GS_SRC"

if [ ! -f Makefile ] || ! grep -q "^CC=/usr/share/emscripten/emcc" Makefile; then
  emconfigure ./configure --host=wasm32-unknown-emscripten \
    --with-drivers=pdfwrite \
    --disable-dynamic \
    --disable-contrib \
    --disable-cups \
    --disable-fontconfig \
    --disable-gtk \
    --without-x \
    --without-libtiff \
    --without-pdftoraster \
    --without-ijs \
    --without-tesseract
fi

emmake make gs -j4 \
  XLDFLAGS="-sALLOW_MEMORY_GROWTH=1 -sINITIAL_MEMORY=67108864 -sEXIT_RUNTIME=1 -sNODERAWFS=1"

cp bin/gs bin/gs.wasm "$POC/"

echo "Build artifacts copied to $POC/"
```

- [ ] **Step 4: Create the verification script**

Create `poc/run.js`:

```javascript
'use strict';

const { execFileSync } = require('child_process');
const fs = require('fs');
const path = require('path');

const POC_DIR = __dirname;
const GS_SRC = path.join(POC_DIR, '..', 'ghostscript-10.07.1');
const GS = path.join(POC_DIR, 'gs');
const INPUT = path.join(POC_DIR, 'test.ps');
const OUTPUT_HOST = path.join(POC_DIR, 'out.pdf');

// -sNODERAWFS=1 makes the wasm binary's getenv/fopen calls operate on the
// real host filesystem, so GS_LIB and -sOutputFile work with normal
// absolute paths exactly as they would for a natively-built gs.
const GS_LIB = [
  path.join(GS_SRC, 'lib'),
  path.join(GS_SRC, 'Resource', 'Init'),
].join(':');

if (fs.existsSync(OUTPUT_HOST)) {
  fs.unlinkSync(OUTPUT_HOST);
}

execFileSync('node', [
  GS,
  '-sDEVICE=pdfwrite',
  '-dNOPAUSE',
  '-dBATCH',
  `-sOutputFile=${OUTPUT_HOST}`,
  INPUT,
], { stdio: 'inherit', env: { ...process.env, GS_LIB } });

if (!fs.existsSync(OUTPUT_HOST)) {
  throw new Error(`Expected output file not found: ${OUTPUT_HOST}`);
}

const header = fs.readFileSync(OUTPUT_HOST, { encoding: 'utf8', flag: 'r' }).slice(0, 4);
if (header !== '%PDF') {
  throw new Error(`Output file does not start with %PDF, got: ${JSON.stringify(header)}`);
}

console.log('PoC OK: poc/out.pdf starts with %PDF');
```
